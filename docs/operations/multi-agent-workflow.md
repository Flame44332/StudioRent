# Multi-Agent Workflow

Дата: 2026-06-07

## Главное правило

Пользователь общается только с главным агентом-оркестратором.

Оркестратор сам решает, когда подключать дополнительных агентов, формулирует им задачи, принимает результаты, проверяет качество и переводит выводы в понятные решения для проекта.

## Роль оркестратора

- Ведет общий контекст проекта.
- Держит продуктовую цель, текущие решения и ограничения.
- Делегирует задачи агентам только когда это ускоряет работу или повышает качество.
- Не перекладывает на пользователя внутреннюю координацию.
- Возвращает пользователю сжатый результат: решения, риски, следующие шаги.

## Базовые агенты

### Product Strategist

Отвечает за MVP, ценность для клиента и студии, бизнес-модель, приоритеты, roadmap.

### UX/UI Designer

Отвечает за пользовательские сценарии, Figma-макеты, дизайн-систему, состояния экранов, Telegram Mini App UX.

### Researcher

Отвечает за рынок студий Москвы, конкурентов, цены, фильтры, пользовательские боли, аналоги и источники.

### Architect

Отвечает за стек, системную архитектуру, базу данных, интеграции, безопасность, платежи, роли и бронирования.

### Coder / Worker

Отвечает за реализацию frontend, backend, Telegram Mini App, API, интеграции и тесты.

### QA / Reviewer

Отвечает за проверку сценариев, визуальные баги, регрессии, крайние случаи, мобильные состояния.

## Как это работает

1. Пользователь ставит задачу оркестратору.
2. Оркестратор определяет, нужна ли параллельная работа.
3. Если нужна, оркестратор запускает одного или нескольких агентов с четкими задачами.
4. Оркестратор параллельно делает критическую работу сам, если она не зависит от агентов.
5. Оркестратор собирает результаты, проверяет их и принимает итоговое решение.
6. Пользователь получает только итог: что решено, что сделано, что дальше.

## Когда подключать агентов

- Нужен быстрый ресерч по нескольким независимым темам.
- Нужно сравнить варианты архитектуры или продукта.
- Нужно параллельно делать дизайн, документацию и код.
- Есть большая задача, которую можно разделить на независимые зоны.
- Нужна отдельная проверка качества после реализации.

## Когда не подключать агентов

- Задача маленькая и быстрее сделать напрямую.
- Следующий шаг критически зависит от одного ответа, который проще получить самому.
- Делегирование создаст больше координации, чем пользы.
- Задача требует единого непрерывного контекста без разделения.

## Формат отчетов пользователю

Оркестратор не показывает весь внутренний шум.

В обычном ответе достаточно:

- что было сделано;
- какие решения приняты;
- какие риски есть;
- что нужно от пользователя;
- какой следующий шаг.
