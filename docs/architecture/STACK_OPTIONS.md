# Stack options

## Вариант A: Next.js + Postgres + Prisma

Плюсы:
- Один проект для frontend и backend routes.
- Хорошо подходит для Telegram Mini App с SSR/CSR гибкостью.
- Prisma быстро дает модели, миграции и типы.
- Легко развернуть MVP.

Минусы:
- При росте backend-логики может стать тесно в одном приложении.
- Нужна дисциплина в разделении API, доменной логики и UI.

Рекомендация: лучший старт для MVP.

## Вариант B: React/Vite + отдельный Node/NestJS backend

Плюсы:
- Четкое разделение frontend и backend.
- NestJS удобен для сложной бизнес-логики, ролей, платежей, уведомлений.
- Проще масштабировать командно.

Минусы:
- Больше инфраструктуры и файлов на старте.
- Медленнее запуск MVP.

Рекомендация: хорошо для v1, если сразу ожидаем сложный кабинет студии.

## Вариант C: React/Vite + Supabase

Плюсы:
- Быстрый старт базы, auth, storage.
- Меньше backend-кода.
- Удобно для прототипов.

Минусы:
- Telegram auth и бизнес-логика бронирований потребуют аккуратной настройки.
- Возможны ограничения по региону, оплатам и контролю данных.

Рекомендация: годится для быстрого прототипа, но для marketplace лучше иметь свой backend-контур.

## Предварительный выбор

Для текущего проекта: `Next.js + TypeScript + Postgres + Prisma`.

Причина: быстро собрать Telegram Mini App, карту, каталог, mock-оплату и первые API, не разводя два отдельных приложения раньше времени.
